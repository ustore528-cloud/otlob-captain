# اختبار يدوي — تطبيق الكابتن (Manual QA)

## 1) إعداد التشغيل على جهاز حقيقي

### الخادم (API)

1. انسخ `apps/api/.env.example` إلى `.env` واضبط `DATABASE_URL` والأسرار.
2. شغّل الـ API من الجهاز المضيف (يجب أن يظهر في السجل: `listening on http://0.0.0.0:4000`).
3. تأكد أن جدار الحماية على Windows يسمح بالوارد على المنفذ `4000` لشبكة LAN (أو عطّل مؤقتًا للاختبار).
4. `CORS_ORIGIN`: الافتراضي الحالي `*` مناسب لتطوير الموبايل على الشبكة المحلية؛ للإنتاج حدّد دومينات الويب صراحةً.

### تطبيق الموبايل (Expo)

1. في `apps/captain-mobile` أنشئ `.env` من `.env.example`.
2. عيّن **`EXPO_PUBLIC_API_URL`** بدون شرطة مائلة أخيرة:
   - **جهاز حقيقي (نفس شبكة Wi‑Fi):** `http://<IP-الحاسوب>:4000` (مثال: `http://192.168.1.15:4000`).
   - **محاكي Android:** غالبًا `http://10.0.2.2:4000`.
   - **محاكي iOS:** غالبًا `http://localhost:4000` يعمل.
3. أعد تشغيل Metro بعد تغيير `.env` (`expo start -c` إن لزم).
4. **`app.json`:** مفعّل `usesCleartextTraffic` (أندرويد) و`NSAllowsLocalNetworking` (iOS) لدعم **HTTP** على الشبكة المحلية أثناء الاختبار.

---

## 2) متغيرات البيئة (موبايل)

| المتغير | الوصف |
|---------|--------|
| `EXPO_PUBLIC_API_URL` | عنوان الـ REST و Socket.IO (نفس الأصل). يُحقن وقت البناء/التشغيل؛ لا يُقرأ من الخادم مباشرة. |

**ملاحظة:** إذا بقي العنوان `localhost` على جهاز حقيقي، سيظهر تحذير في الـ console ولن تصل الطلبات للحاسوب.

---

## 3) نقاط تفشل غالبًا على الهاتف الحقيقي — ما تمت معالجته في الكود

| الموضوع | السلوك / الإصلاح |
|---------|------------------|
| **HTTP على LAN** | تم تمكين cleartext على أندرويد و`NSAllowsLocalNetworking` على iOS في `app.json`. |
| **CORS + Socket من الموبايل** | افتراضي `CORS_ORIGIN=*` في الـ API لتفادي رفض الطلبات/الـ handshake من أصول غير متوقعة أثناء التطوير. |
| **الاستماع على الشبكة** | الـ API يستمع على `0.0.0.0` حتى يكون قابلاً للوصول من أجهزة أخرى على LAN. |
| **localhost بالخطأ** | تحذير `console.warn` في `env.ts` عند استخدام loopback على منصة أصلية. |

## 4) سلوك متوقع (بدون إشعارات دفع)

| الميزة | الواقع في المشروع |
|--------|-------------------|
| **إشعارات Push (FCM/APNs)** | غير مدمجة حاليًا؛ قائمة الإشعارات تعتمد على **REST** (`/api/v1/notifications`). |
| **Deep links** | المخطط `captain` في `app.json`؛ مسار تفاصيل الطلب تحت `/(app)/order/[orderId]`. |
| **استمرار الجلسة** | التوكنات في `expo-secure-store`؛ عند فشل `GET /me` بسبب **شبكة/خادم مؤقت** لا تُمسح الجلسة؛ تُمسح عند **401 / 403 / 404** من `me`. |

## 5) Checklist اختبار يدوي (مفصّلة)

### أ — تثبيت وتشغيل

- [ ] `EXPO_PUBLIC_API_URL` يشير إلى IP صحيح؛ لا تحذير localhost على جهاز حقيقي.
- [ ] `GET /health` من متصفح الهاتف: `http://<IP>:4000/health` يعيد JSON ناجح.
- [ ] التطبيق يفتح دون شاشة بيضاء دائمة (يختفي الـ splash بعد `sessionReady`).

### ب — المصادقة واستمرار الجلسة (Auth persistence)

- [ ] تسجيل دخول كابتن صالح → الوصول إلى التبويبات.
- [ ] إغلاق التطبيق بالكامل (من multitasking) وإعادة الفتح → لا يُطلب تسجيل الدخول من جديد.
- [ ] «تسجيل الخروج» → يعود لشاشة الدخول ولا تبقى بيانات حساسة ظاهرة.

### ج — الشبكة وعنوان الـ API (Network base URL)

- [ ] لوحة الكابتن تحمّل بيانات `me` بدون خطأ.
- [ ] تعطيل Wi‑Fi لثوانٍ ثم إعادته → الطلبات تعود (سحب للتحديث حيث يتوفر).
- [ ] مع خادم متوقف: رسالة خطأ واضحة وليس تجمّد صامت.

### د — الصلاحيات — الموقع (Location)

- [ ] شاشة التتبع: رفض الإذن → رسالة/حالة واضحة؛ لا كراش.
- [ ] من إعدادات النظام منح الإذن → يمكن طلب الموقع (حسب منطق الشاشة).
- [ ] (اختياري) سلوك الخلفية لاحقًا — غير مطلوب للـ beta إن لم يُفعّل بعد.

### هـ — Socket (اتصال لحظي)

- [ ] بعد تسجيل الدخول، لا أخطاء متكررة في الـ console متعلقة بـ socket (ما لم يكن الخادم مغلقًا).
- [ ] عند إيقاف الخادم ثم تشغيله: إعادة الاتصال تلقائيًا أو تحديث بعد سحب/تنقل بين الشاشات.
- [ ] عند وجود تعيين/طلب من لوحة التحكم: استقبال تحديث أو تحديث بعد fallback polling (نحو 22 ثانية كحد أقصى عند انقطاع الـ socket).

### و — Deep links

- [ ] فتح رابط يستخدم المخطط `captain` لطلب معروف (مثال تجريبي):  
  `captain://order/<orderId>` أو حسب ما يولّده `expo-linking` لنفس المسار.
- [ ] عند فتح الرابط والجلسة صالحة: تُعرض شاشة تفاصيل الطلب أو رسالة منطقية إن كان الطلب غير مسموح.

### ز — الإشعارات (داخل التطبيق فقط)

- [ ] تبويب الإشعارات يحمّل القائمة من الـ API.
- [ ] إن وُجد `orderId` على عنصر، الضغط يفتح تفاصيل الطلب.

### ح — دون اتصال / إعادة اتصال (Offline / reconnect)

- [ ] وضع الطيران ON → محاولة تنفيذ إجراء يظهر خطأ أو حالة تخطيط واضحة.
- [ ] إعادة الشبكة → إعادة المحاولة يدويًا أو تلقائيًا تعيد التزامن دون الحاجة لإعادة تثبيت التطبيق.

### ط — إضافية (Regression سريع)

- [ ] تغيير التوفر (متاح/غير متاح) من الرئيسية أو الملف الشخصي.
- [ ] سجل الطلبات + فتح طلب → تفاصيل + أزرار الحالة إن انطبقت.
- [ ] ملخص الأرباح يعرض أرقامًا أو حالة فارغة منطقية.

## 6) بعد الاختبار

- سجّل: الجهاز (iOS/Android)، إصدار التطبيق، عنوان `EXPO_PUBLIC_API_URL` (بدون أسرار)، ولقطات أو وصف للأعطال.
